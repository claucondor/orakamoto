## 2026-01-20: Implemented Deposit Idle Funds to Yield Source

### Summary
Implemented `deposit-idle-funds` functionality that moves 90% of pool liquidity to the yield source (mock-zest-vault) to earn yield on idle funds. This feature is available in both market-pool (binary prediction markets) and multi-outcome-pool (LMSR-based markets).

### Changes Made

1. **contracts/market-pool.clar**:
   - Added new error constants:
     - `ERR-ALREADY-DEPOSITED` (u1016) - Prevents multiple deposits
     - `ERR-INSUFFICIENT-IDLE-LIQUIDITY` (u1017) - Ensures sufficient liquidity for deposit
   - Added `deposited-to-yield` data variable to track amount deposited to yield source
   - Implemented `deposit-idle-funds` function:
     - Only callable by market creator
     - Only callable before market resolution
     - Moves 90% of total reserves (yes + no) to mock-zest-vault
     - Reduces both yes-reserve and no-reserve proportionally
     - Tracks deposited amount for future withdrawals
   - Implemented `withdraw-yield-funds` function:
     - Only callable by market creator
     - Withdraws all deposited funds (including earned yield) from vault
     - Returns funds to pool reserves (split 50/50 between yes/no)
     - Resets deposited tracking to 0
   - Added read-only functions:
     - `get-deposited-to-yield`: Returns amount currently deposited to yield source
     - `get-available-liquidity`: Returns breakdown of reserves and deposited amount

2. **contracts/multi-outcome-pool.clar**:
   - Added new error constants:
     - `ERR-ALREADY-DEPOSITED` (u2018)
     - `ERR-INSUFFICIENT-IDLE-LIQUIDITY` (u2019)
   - Added `deposited-to-yield` data variable
   - Implemented `deposit-idle-funds` function:
     - Same logic as market-pool but operates on `total-liquidity`
     - Only callable by market creator before resolution
     - Moves 90% of total liquidity to yield source
   - Implemented `withdraw-yield-funds` function:
     - Withdraws deposited funds back to total liquidity
     - Includes earned yield in returned amount
   - Added read-only functions:
     - `get-deposited-to-yield`
     - `get-available-liquidity`

3. **Clarinet.toml**:
   - Updated `market-pool` to depend on `mock-zest-vault`
   - Updated `multi-outcome-pool` to depend on `mock-zest-vault`

4. **tests/deposit-idle-funds.test.ts** (new file):
   - Comprehensive tests for market-pool deposit-idle-funds:

## 2026-01-20: Implemented Multi-Outcome Market Pool with LMSR

### Summary
Created the multi-outcome-pool.clar contract implementing LMSR (Logarithmic Market Scoring Rule) for markets with 2-10 outcomes. This enables prediction markets beyond binary YES/NO outcomes.

### Changes Made

1. **contracts/multi-outcome-pool.clar** (new file):
   - Implemented LMSR math using integer approximations:
     - `exp-approx`: Taylor series expansion for e^x
     - `ln-approx`: Newton-Raphson approximation for ln(x)
   - Added support for 2-10 outcomes per market
   - Implemented LMSR cost function: Cost(q) = b * ln(sum(exp(q_i / b)))
   - Implemented price calculation: Price_i = exp(q_i / b) / sum(exp(q_j / b))
   - Added `outcome-count` and `lmsr-b` (liquidity parameter) data variables
   - Implemented `initialize` function with configurable outcome count and LMSR b parameter
   - Implemented `buy-outcome` and `sell-outcome` using LMSR pricing
   - Implemented `resolve`, `claim`, `open-dispute`, `finalize-resolution` (same as binary markets)
   - Added comprehensive error constants (ERR-INVALID-OUTCOME-COUNT, ERR-OUTCOME-NOT-RESOLVED)
   - Note: Does NOT implement prediction-market-trait (designed for multi-outcome markets)

2. **contracts/traits/prediction-market-trait.clar**:
   - Updated to clarify that multi-outcome-pool is a separate implementation

3. **tests/multi-outcome-pool.test.ts** (new file):
   - Tests for LMSR math functions (exp-approx, ln-approx)
   - Tests for market initialization with 3-4 outcomes
   - Tests for validation (invalid outcome count, insufficient collateral)
   - Tests for add-liquidity functionality
   - Tests for buy/sell outcome tokens
   - Tests for market resolution and claims

## 2026-01-20: Implemented Yield Distributor Contract

### Summary
Created the yield-distributor.clar contract that handles yield distribution logic for LPs based on time-weighted LP balances. This contract provides the infrastructure for LPs to claim their share of yield earned by the protocol.

### Changes Made

1. **contracts/yield-distributor.clar** (new file):
   - Implemented SIP-010 fungible token interface (yLD - yield distributor token)
   - Added LP balance tracking with time-weighted updates
   - Implemented `update-lp-balance` function:
     - Called by pool contracts when LP tokens are minted/burned
     - Tracks LP balance with timestamp for time-weighted calculations
     - Only callable by the pool contract (authorization via contract-caller)
   - Implemented `deposit-yield` function:
     - Called by yield vaults to deposit earned yield for distribution
     - Mints yLD tokens proportional to yield amount
     - Tracks pool-specific yield and total accumulated yield
     - Only callable by authorized yield sources (yield-vault, mock-zest-vault)
   - Implemented `claim-yield` function:
     - Allows LPs to claim their share of accumulated yield
     - Calculates proportional share based on time-weighted LP balance
     - Mints yLD tokens to claimer as yield claim receipts
     - Prevents double-claiming via tracking
   - Implemented `calculate-pending-yield` function:
     - Calculates pending yield for a specific LP in a pool
     - Returns 0 if already claimed
   - Added comprehensive read-only functions:
     - `get-total-yield-accumulated`: Total yield across all pools
     - `get-total-claims`: Total yield claimed by LPs

     - `get-lp-time-weighted-balance`: Time-weighted LP balance
     - `get-pending-yield`: User's pending yield across all pools
     - `get-has-claimed-yield`: Check if user claimed yield for pool
   - Added admin functions for testing:
     - `mint`: Mint yLD tokens (owner only)
     - `burn`: Burn yLD tokens (token holder)
   - Events emitted for all actions
   - Error codes: ERR-NOT-AUTHORIZED (u300), ERR-NOT-TOKEN-OWNER (u301), ERR-ZERO-AMOUNT (u302), ERR-INSUFFICIENT-BALANCE (u303), ERR-ALREADY-CLAIMED (u304), ERR-NO-YIELD-AVAILABLE (u305), ERR-NOT-INITIALIZED (u307)
