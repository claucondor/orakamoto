## 2026-01-21: Implemented Reputation Registry Contract (Phase 6 - Hybrid Reputation Oracle)

### Summary
Created the reputation-registry.clar contract that implements Layer 4 (Quadratic Reputation Voting) for the Hybrid Reputation Oracle (HRO). This contract tracks voter reputation based on historical accuracy and participation, enabling quadratic voting power calculation to limit whale influence.

### Changes Made

1. **contracts/reputation-registry.clar** (new file):
   - Implements SIP-010 fungible token ($PRED) with 8 decimals
   - Reputation score tracking per voter:
     - `correct-votes`: Number of votes that matched final outcome
     - `total-votes`: Total number of votes cast
     - `participation-score`: Participation rate (0-1000000 = 0-100%)
     - `last-updated`: Block height when reputation was last updated
     - `total-earned`: Total $PRED tokens earned
   - Reputation score calculation: `(correct_votes / total_votes) × participation_rate × PRECISION`
   - Quadratic vote power calculation: `√(tokens_staked) × reputation_score × time_multiplier`
   - Time multiplier: 1.0x to 4.0x based on lock duration (1 week to 4 years)
   - Decay function: 1% reputation decay per month of inactivity (43200 blocks)
   - Reputation history tracking for transparency/auditing
   - Vote tracking to prevent double voting on same market
   - Integer square root approximation using Babylonian method

   **Key Functions:**
   - `get-reputation-score`: Calculate current reputation score (0-1000000)
   - `calculate-vote-power`: Calculate quadratic voting power
   - `record-vote-cast`: Track when a voter casts a vote (called by quadratic-voting)
   - `update-reputation`: Update reputation after vote resolution (called by quadratic-voting)
   - `apply-decay`: Apply reputation decay for inactivity
   - `get-decayed-reputation`: Get reputation with decay applied
   - `has-voted-on-market`: Check if voter already voted on a market

   **Read-Only Functions:**
   - `get-reputation`: Get full reputation record
   - `get-participation-rate`: Get participation rate for voter
   - `get-total-votes`: Get total votes cast by voter
   - `get-correct-votes`: Get correct votes by voter
   - `get-total-earned`: Get total $PRED earned
   - `get-reputation-history`: Get reputation history by ID
   - `get-history-count`: Get total history count

   **Admin Functions:**
   - `initialize-reputation`: Initialize reputation for a voter (owner only)
   - `reset-reputation`: Reset reputation for a voter (owner only)

2. **Clarinet.toml**:
   - Added reputation-registry contract configuration with sip-010-trait dependency

3. **tests/reputation-registry.test.ts** (new file):
   - 50+ comprehensive tests covering:
     - SIP-010 metadata (name, symbol, decimals, supply, balance)
     - Mint functionality with access control
     - Transfer functionality
     - Burn functionality
     - Reputation initialization with validation
     - Reputation score calculation (perfect accuracy, zero accuracy, new voters)
     - Vote power calculation with quadratic formula
     - Time multiplier calculation (1 week to 4 years, capped at 4.0x)
     - Vote recording with authorization checks
     - Double voting prevention
     - Reputation update after vote resolution
     - Reputation decay (1 month, multiple months, minimum bounds)
     - Decayed reputation calculation
     - Reputation history tracking
     - Market vote tracking
     - Reset reputation functionality
     - Edge cases (perfect accuracy, zero accuracy, large token amounts)

### Design Notes
- Follows the same patterns as existing governance contracts (vote-escrow, governance-token)
- Uses `contract-caller` for authorization (secure against phishing)
- Reputation score is calculated dynamically from correct_votes / total_votes
- Participation score decays over time (1% per month of inactivity)
- Quadratic voting formula limits whale influence: √(tokens) × reputation × time_multiplier
- Time multiplier rewards long-term commitment (1x to 4x)
- Integer square root approximation avoids floating-point math
- All reputation updates are recorded in history for transparency
- Minimum reputation score of 0.1 (100000) for new voters
- Maximum reputation score of 1.0 (1000000) for perfect accuracy

## 2026-01-20: Implemented Creator Rewards Contract (Phase 5 - Governance)

## 2026-01-20: Implemented Oracle Resolver (Phase 4 - Auto-Resolution System)

### Summary
Created the oracle-resolver.clar contract that enables markets to be automatically resolved via oracle price feeds. This implements the auto-resolution system for Phase 4 (Oracle Integration) with support for three resolution types: MANUAL, PRICE_TARGET, and TIME_BASED.

### Changes Made

1. **contracts/oracle-resolver.clar** (new file):
   - Added resolution type constants:
     - `RESOLUTION-TYPE-MANUAL` (u0) - Creator must resolve manually
     - `RESOLUTION-TYPE-PRICE-TARGET` (u1) - Auto-resolve when oracle price meets target
     - `RESOLUTION-TYPE-TIME-BASED` (u2) - Auto-resolve based on oracle price at deadline
   - Added error constants:
     - `ERR-NOT-AUTHORIZED` (u4100)
     - `ERR-MARKET-NOT-FOUND` (u4101)
     - `ERR-INVALID-RESOLUTION-TYPE` (u4102)
     - `ERR-INVALID-TARGET-PRICE` (u4103)
     - `ERR-INVALID-ORACLE` (u4104)
     - `ERR-ALREADY-CONFIGURED` (u4105)
     - `ERR-RESOLUTION-NOT-TRIGGERED` (u4106)
     - `ERR-PRICE-STALE` (u4107)
     - `ERR-PRICE-NOT-AVAILABLE` (u4108)
     - `ERR-MARKET-NOT-READY` (u4109)
   - Implemented `oracle-market-config` map storing:
     - `oracle-contract`: Optional principal implementing oracle-trait
     - `price-feed-id`: Asset identifier (e.g., "BTC", "ETH")
     - `target-price`: Price threshold for PRICE_TARGET resolution (8 decimals)
     - `resolution-type`: 0 = MANUAL, 1 = PRICE_TARGET, 2 = TIME_BASED
     - `max-price-age`: Maximum allowed age of oracle price in blocks
     - `is-active`: Whether auto-resolution is enabled
     - `resolved`: Whether the market has been auto-resolved
     - `resolved-outcome`: The outcome that was resolved (0 = YES, 1 = NO)
   - Implemented `configure-market` function:
     - Allows configuring a market for oracle-based auto-resolution
     - Validates resolution type and required parameters
     - Only callable once per market
   - Implemented `check-resolution` function:
     - Anyone can call to trigger auto-resolution
     - Checks if resolution conditions are met
     - Marks market as resolved with winning outcome
   - Implemented `can-auto-resolve` function:
     - Returns whether market can be auto-resolved
     - Provides reason and winning outcome if applicable
   - Implemented `deactivate-auto-resolution` function:
     - Allows deactivating auto-resolution for a market
   - Implemented `update-oracle-config` function:
     - Allows owner to update oracle configuration
   - Implemented `reset-resolution` function:
     - Allows owner to reset resolution status (for testing/correction)
   - Implemented helper functions:
     - `get-current-price`: Get current price from oracle
     - `is-price-fresh-for-market`: Check if price is fresh for market
     - `get-oracle-config`: Get configuration for a market
   - Events emitted for all actions (configure, auto-resolve, deactivate, update, reset)

2. **Clarinet.toml**:
   - Added oracle-resolver contract configuration with oracle-trait dependency

3. **tests/oracle-resolver.test.ts** (new file):
   - 33 comprehensive tests covering:
     - Resolution type constants
     - Market configuration for PRICE_TARGET, TIME_BASED, and MANUAL resolution
     - Validation (invalid resolution type, zero target price, missing oracle)
     - Reject configuring already configured market
     - Get oracle config for configured and non-existent markets
     - Get current price from oracle
     - Price freshness checks (fresh and stale prices)
     - Check resolution - PRICE_TARGET:
       - Resolve to YES when price >= target
       - Resolve to NO when price < target
       - Reject resolution if price is stale
       - Reject resolution if market already resolved
     - Check resolution - MANUAL:
       - Reject resolution for MANUAL type
     - Deactivate auto-resolution:
       - Allow deactivating
       - Reject deactivating already inactive market
     - Update oracle config:
       - Allow owner to update
       - Reject update from non-owner
       - Reject update for inactive market
     - Reset resolution:
       - Allow owner to reset
       - Reject reset from non-owner
     - Can auto resolve:
       - Return can-resolve=true for PRICE_TARGET when price meets target
       - Return can-resolve=false for MANUAL type
       - Return can-resolve=false for non-configured market
     - Integration tests:
       - Full lifecycle: configure -> check -> resolve
       - Handle multiple markets independently
       - Handle price updates between configuration and resolution

### Design Notes
- The oracle-resolver uses the oracle-trait interface to be swappable between different oracle providers
- PRICE_TARGET resolution: YES wins if oracle price >= target, NO wins otherwise
- TIME_BASED resolution: Ready to resolve when market deadline passes (uses oracle price at deadline)
- MANUAL resolution: Requires creator to resolve manually (no auto-resolution)
- Price staleness is checked against max-price-age parameter (default: 1008 blocks ~7 days)
- The contract integrates with existing market infrastructure via the oracle-market-config map
- Auto-resolution can be triggered by anyone (permissionless)

### Design Notes

- **Economic Security**: Each challenge requires doubling the bond, making spam attacks expensive

- **Permissionless**: Anyone can initiate disputes and trigger finalization

- **Devnet compatible**: Uses local trait reference for USDC token

### Design Notes
- Follows the same patterns as existing governance contracts (vote-escrow, dispute)
- Uses `contract-caller` for authorization (secure against phishing)
- Commit-reveal scheme prevents last-minute vote manipulation:
  - Voters commit hash before knowing others' votes
  - Reveals happen after commit phase ends
  - Non-revealers lose 10% of staked tokens
- Quadratic voting formula: `√(tokens) × reputation × time_multiplier`
  - Limits whale influence: 100x tokens = 10x voting power (not 100x)
  - Reputation from reputation-registry (historical accuracy + participation)
  - Time multiplier: 1x to 4x based on lock duration
- Session lifecycle:
  1. HRO bond threshold reached → create voting session
  2. Commit phase (2 days): Voters commit and stake tokens
  3. Reveal phase (1 day): Voters reveal votes
  4. Tally phase: Count votes with quadratic weighting
  5. Update reputation for voters
  6. Distribute rewards to winning voters

### Integration with HRO
- `hro-resolver.trigger-voting()` emits event when bond threshold reached
- Frontend listens for event and calls `create-voting-session()`
- After tally, `update-voter-reputation()` updates voter reputation in reputation-registry
- Bond distribution happens after reputation updates
