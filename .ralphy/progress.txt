### Design Notes

**Parallel Query Strategy:**
- Uses `Promise.all()` to query all models simultaneously
- Failed queries are filtered out (don't block aggregation)
- Models queried: GPT-4 Turbo, Claude 3 Opus, Llama 3 70B (default)

**Confidence-Weighted Voting:**
- Each model has a default weight based on expected accuracy
- Higher confidence = more weight in final calculation
- Weighted YES votes vs Weighted NO votes determines outcome
- Consensus % = (max(weightedYes, weightedNo) / totalWeight) * 100

**Aggregation Logic:**
```
weightedYes = Σ(modelVote == YES ? confidence * weight : 0)
weightedNo = Σ(modelVote == NO ? confidence * weight : 0)
totalWeight = weightedYes + weightedNo
yesPercentage = (weightedYes / totalWeight) * 100
noPercentage = (weightedNo / totalWeight) * 100
outcome = yesPercentage >= noPercentage ? YES : NO
consensus = max(yesPercentage, noPercentage)
```

**Consensus Threshold:**
- Default: 60% minimum consensus to submit
- Can be customized with --min-consensus flag
- Auto-submit with --auto flag when threshold met
- Below threshold: Shows warning but doesn't submit

**Contract Integration:**
- Calls `request-ai-evaluation` to initiate evaluation
- In production, would call `record-ai-recommendation` for each model
- Requires models to be registered with model-id in contract
- Evidence links limited to 10 URLs max (200 chars each)

**Model Selection:**
- OpenRouter (recommended): Single API key for all models
- Direct API: OpenAI for GPT-4, Anthropic for Claude
- Fallback logic: Try OpenRouter first, fall back to direct API

- Transaction broadcasting errors with clear error messages
- Rate limiting (500ms delay between assets)

**Watch Mode:**
- Updates every 60 seconds (Pyth publishes price updates ~400ms)
- Graceful exit on Ctrl+C (clears interval)
- Suitable for 24/7 operation with pm2 or similar process manager

### Design Notes

**Assertion API Patterns:**
- Use `.toBeOk(VALUE)` to assert ok response with expected value
- Use `.toBeErr(Cl.uint(ERROR_CODE))` to assert error response
- For tuple values: access as `(result.result as any).value.value['field-name'].value`
- For list values: access as `(result.result as any).value.value` (the value property contains the array)

**LP Token Authorization:**
- Must be set up in both `beforeAll` AND `beforeEach` to persist across tests
- Use `ensureLpTokenSetup()` pattern that checks current state before setting

**Contract Error Wrapping:**
- Error constants defined as `(err uXXXX)` should NOT be wrapped in another `(err ...)` in match expressions
- Use the error constant directly in the else branch of match expressions

**Clarity Type System Limitations:**
- String UTF8 max length and List max length are enforced at TYPE level, not runtime
- Cannot catch these errors in contract code - tests should expect runtime errors for these cases

## 2026-01-21: Governance Attack Scenario Tests (Phase 5 - Governance)

### Summary
Created comprehensive test suite for governance attack scenarios in tests/governance.test.ts, covering flash loan voting protection, last-minute swing attacks, proposal spam protection, and other governance vulnerabilities.

### Changes Made

1. **tests/governance.test.ts** (extended):
   - Added "Attack Scenario Tests" section with 20+ new tests covering:

   **Proposal Spam Protection:**
   - Prevents rapid proposal creation by enforcing cooldown period (1 proposal per week per address)
   - Allows different addresses to create proposals during cooldown

   **Vote Manipulation Protection:**
   - Prevents double voting on same proposal (ERR-ALREADY-VOTED)
   - Counts votes correctly even with coordinated voting timing

   **Last-Minute Voting Swings:**
   - Allows voting until voting period ends (no front-running protection needed since votes are public)
   - Rejects voting after voting period ends (ERR-PROPOSAL-NOT-ACTIVE)
   - Requires timelock to pass before execution (prevents instant execution)
   - Only executes after both voting period and timelock end

   **Minimal Voting Power Threshold Protection:**
   - Prevents proposal creation without minimum voting power threshold
   - Emergency proposals require 10x higher threshold

   **Quorum and Majority Protection:**
   - Prevents execution without reaching quorum (ERR-QUORUM-NOT-REACHED)
   - Prevents execution without majority approval

   **Emergency Proposal Protection:**
   - Validates emergency proposal thresholds (10x normal threshold)

   **Social Engineering Protection (tx-sender vs contract-caller):**
   - Verifies admin functions use contract-caller for authorization
   - Prevents phishing attacks via tx-sender manipulation

   **Token Supply Manipulation Protection:**
   - Prevents minting without owner authorization
   - Prevents burning more than balance

   **Parameter Update Protection:**
   - Prevents non-owner from updating governance parameters
   - Validates parameter bounds (e.g., fee cannot exceed 100%)

   **Collusion and Sybil Attack Scenarios:**
   - Documents that multiple coordinated addresses can vote independently (decentralized system feature)
   - No mechanism to prevent legitimate multi-address voting (by design)

   **Proposal Cancellation Protection:**
   - Prevents non-proposer from canceling proposal
   - Allows proposer to cancel before timelock ends
   - Prevents cancellation after timelock ends

### Key Security Features Verified

1. **Flash Loan Voting Protection:**
   - Voting power is calculated from locked tokens in vote-escrow
   - Cannot borrow, vote, and repay in one transaction
   - Tokens must be locked for minimum 1 week to get voting power
   - Lock duration affects voting power (longer locks = more power)

2. **Last-Minute Swing Protection:**
   - Commit-reveal scheme in quadratic-voting prevents last-minute manipulation
   - Voters commit hash before knowing others' votes
   - Reveals happen after commit phase ends
   - Non-revealers lose 10% of staked tokens

3. **Proposal Spam Protection:**
   - Cooldown period: 1 proposal per week per address
   - Minimum voting power threshold required to create proposals
   - Emergency proposals require 10x threshold

4. **Quorum and Majority Requirements:**
   - Quorum: 10% of total voting power must participate
   - Majority: >50% approval for normal proposals
   - Emergency: 80% approval required

5. **Timelock Protection:**
   - Voting period: 7 days (1008 blocks)
   - Timelock: 2 days (288 blocks) after voting ends
   - Prevents instant execution even with majority

### Design Notes

- **No Flash Loan Protection Needed:** In Clarity/Stacks, flash loans are not possible in the same way as Ethereum since contracts cannot call back into themselves during execution. The vote-escrow mechanism (locking tokens for weeks) provides natural protection.

- **Last-Minute Voting:** The system allows last-minute voting because:
  - Votes are public on-chain (no secret voting)
  - Commit-reveal scheme in quadratic-voting prevents manipulation
  - In a decentralized system, voters have right to vote until deadline

- **Sybil Attacks:** Cannot be fully prevented in a decentralized system. The quadratic voting formula (√(tokens) × reputation) limits whale influence but doesn't prevent multiple addresses.

- **Social Engineering:** Protected by using `contract-caller` instead of `tx-sender` for authorization in all admin functions.

## 2026-01-21: Implemented Reputation Registry Contract (Phase 6 - Hybrid Reputation Oracle)

### Summary
Created the reputation-registry.clar contract that implements Layer 4 (Quadratic Reputation Voting) for the Hybrid Reputation Oracle (HRO). This contract tracks voter reputation based on historical accuracy and participation, enabling quadratic voting power calculation to limit whale influence.

### Changes Made

1. **contracts/reputation-registry.clar** (new file):
   - Implements SIP-010 fungible token ($PRED) with 8 decimals
   - Reputation score tracking per voter:
     - `correct-votes`: Number of votes that matched final outcome
     - `total-votes`: Total number of votes cast
     - `participation-score`: Participation rate (0-1000000 = 0-100%)
     - `last-updated`: Block height when reputation was last updated
     - `total-earned`: Total $PRED tokens earned
   - Reputation score calculation: `(correct_votes / total_votes) × participation_rate × PRECISION`
   - Quadratic vote power calculation: `√(tokens_staked) × reputation_score × time_multiplier`
   - Time multiplier: 1.0x to 4.0x based on lock duration (1 week to 4 years)
   - Decay function: 1% reputation decay per month of inactivity (43200 blocks)
   - Reputation history tracking for transparency/auditing
   - Vote tracking to prevent double voting on same market
   - Integer square root approximation using Babylonian method

   **Key Functions:**
   - `get-reputation-score`: Calculate current reputation score (0-1000000)
   - `calculate-vote-power`: Calculate quadratic voting power
   - `record-vote-cast`: Track when a voter casts a vote (called by quadratic-voting)
   - `update-reputation`: Update reputation after vote resolution (called by quadratic-voting)
   - `apply-decay`: Apply reputation decay for inactivity
   - `get-decayed-reputation`: Get reputation with decay applied
   - `has-voted-on-market`: Check if voter already voted on a market

   **Read-Only Functions:**
   - `get-reputation`: Get full reputation record
   - `get-participation-rate`: Get participation rate for voter
   - `get-total-votes`: Get total votes cast by voter
   - `get-correct-votes`: Get correct votes by voter
   - `get-total-earned`: Get total $PRED earned
   - `get-reputation-history`: Get reputation history by ID
   - `get-history-count`: Get total history count

   **Admin Functions:**
   - `initialize-reputation`: Initialize reputation for a voter (owner only)
